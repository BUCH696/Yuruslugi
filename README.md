# Yuruslugi Landing

Строгий одностраничный сайт для юридических услуг с сохранением заявок, подключением аналитики и простой эксплуатацией на VDS.

## Что уже сделано

- Лендинг без тяжелого фронтенд-сборщика.
- API для формы заявок на `Express`.
- Сохранение лидов в `SQLite` с UTM-метками, реферером и URL страницы.
- Подключение `Яндекс.Метрики` и `Google Tag Manager` через `.env`.
- Базовые меры защиты: `helmet`, сжатие, rate limit, healthcheck.
- Файлы для запуска через `systemd`, `nginx` и `Docker`.

## Структура

```text
.
├── public/              # Лендинг: HTML, CSS, JS
├── src/                 # Сервер, конфиг, база
├── deploy/              # Примеры конфигов для VDS
├── Dockerfile
├── .env.example
└── package.json
```

## Локальный запуск

1. Установить зависимости:

```bash
npm install
```

2. Создать `.env`:

```bash
cp .env.example .env
```

3. Запустить проект:

```bash
npm start
```

Сайт будет доступен на `http://localhost:3000`.

## Переменные окружения

- `PORT` и `HOST`: адрес и порт сервера.
- `SITE_URL`: публичный URL сайта.
- `SITE_NAME`, `SITE_PHONE`, `SITE_EMAIL`, `SITE_ADDRESS`: контактные данные на лендинге.
- `DATABASE_PATH`: путь к файлу `SQLite`.
- `YANDEX_METRIKA_ID`: идентификатор счетчика Яндекс.Метрики.
- `GOOGLE_TAG_MANAGER_ID`: контейнер GTM.
- `REQUEST_RATE_LIMIT`: лимит POST-запросов к API в минуту.

## Архитектура для VDS

Для одного VDS разумная схема такая:

1. `nginx` принимает внешний трафик, завершает TLS и проксирует в Node.
2. Node-приложение работает как `systemd`-сервис с `Restart=always`.
3. `SQLite` хранится на локальном SSD в режиме `WAL`.
4. Регулярный backup базы уходит по `cron`.
5. Логи приложения и `nginx` мониторятся отдельно.

Это не полноценная high-availability архитектура, потому что один VDS остается единой точкой отказа. На одном сервере можно повысить устойчивость, но нельзя получить настоящую отказоустойчивость уровня HA.

## Что рекомендую для максимальной устойчивости

Если планируется активный рекламный трафик и реальные заявки:

1. Оставить текущую версию как стартовую.
2. При росте обращений перевести базу с `SQLite` на `PostgreSQL`.
3. Делать резервные копии не только локально, но и во внешнее хранилище.
4. Вынести мониторинг в `Uptime Kuma`, `Zabbix` или внешний uptime-check.
5. Для реальной отказоустойчивости поднять второй сервер или использовать managed database.

## Что собирать в рекламе и аналитике

Форма уже сохраняет:

- имя;
- телефон;
- выбранную услугу;
- комментарий;
- `utm_source`;
- `utm_medium`;
- `utm_campaign`;
- `utm_term`;
- `utm_content`;
- URL страницы;
- `referrer`.

Этого достаточно для базовой оценки эффективности Яндекс.Директа, Google Ads и сторонних каналов.

## Деплой на Linux VDS

1. Скопировать проект в `/opt/yuruslugi`.
2. Выполнить `npm install --omit=dev`.
3. Создать рабочий `.env`.
4. Скопировать `deploy/yuruslugi.service` в `/etc/systemd/system/`.
5. Скопировать `deploy/nginx.conf` в конфигурацию `nginx`, настроить домен и SSL.
6. Включить сервис:

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now yuruslugi
sudo systemctl status yuruslugi
```

7. Настроить резервное копирование `deploy/backup-sqlite.sh` через `cron`.
